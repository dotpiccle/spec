# Non-normative numeric aids

These values help engine authors catch transcription and boundary mistakes without embedding a reference engine or PCM output. The normative formulas remain in `docs/`; these aids alone do not prove audible conformance.

Although these aids are labeled non-normative as format definition, they are **normatively referenced** by the conformance gates in [Engine Build Guide](../../docs/15-engine-build-guide.md). A conforming engine MUST recompute and compare every entry. Deterministic entries match exactly. The `balance` and `lowpass_1000_hz_48000_resonance_0` fields use the portable binary64 tolerance defined in build-guide step 3 because their `sin`/`cos` results may vary by a few last bits across processors. If an aid disagrees with the formulas in [the normative documents](../../docs/), the formulas are authoritative; see [Conformance](../../docs/14-conformance.md) §Role of repository fixtures.

[dsp-values.json](dsp-values.json) covers PCG32 initialization, transition-curve checkpoints, oscillator Fourier coefficients and DFT convention, a canonical biquad coefficient set, equal-power balance and mono adaptation, absolute timing boundaries, pitch transformation order, canonical mixing order, render-profile frequency clamping, and reverb tail boundaries, terminal windows, and baseline generator configuration. [reverb-matrix-vector.json](reverb-matrix-vector.json) contains a test vector for the random orthogonal feedback matrix construction. [reverb-qualification-matrix.json](reverb-qualification-matrix.json) contains the mandatory 10 document-configuration triples, the finite representative sample-rate set for continuous-rate render profiles, and the recommended property-test domain (see [Engine Build Guide](../../docs/15-engine-build-guide.md) step 6). [reverb-reference-irs/](reverb-reference-irs/) contains the canonical reference IR render fixtures for the perceptual-equivalence gate.
